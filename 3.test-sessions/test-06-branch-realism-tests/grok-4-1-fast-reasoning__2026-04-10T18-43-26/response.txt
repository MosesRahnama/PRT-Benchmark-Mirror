**No, the helper strategy is unsound.**

The first theorem `kappa_rec_delta_step` claims ∀ b s n, `kappa (recDelta b s (delta n)) = kappa (recDelta b s n) + 1`, but this fails when `n` structurally begins with `delta`.

**Counterexample:** Let `b := void`, `s := void`, `n := delta void`.
- `kappa void = 0`
- `kappa (delta void) = 0`
- LHS: `kappa (recDelta void void (delta (delta void))) = max(max(0, 0), 0) + 1 = 1`
- RHS: `kappa (recDelta void void (delta void)) = max(max(0, 0), 0) + 1 = 1`
- `1 ≠ 1 + 1`

The `kappa` definition uses pattern matching: `recDelta b s (.delta n)` adds `+1`; the catch-all does not. When `n = delta k`, both sides add `+1`, violating the equality.

The second theorem relies on the first (via `rw [kappa_rec_delta_step]` in its proof sketch), so it inherits the issue. Even if the first held, the second would be sound (`kappa (app s (recDelta b s n)) = max(kappa s, kappa (recDelta b s n)) ≤ kappa (recDelta b s n) < kappa (recDelta b s n) + 1`), but the strategy fails as stated.
