## Case 2 — `R_rec_zero`: `recDelta b s void → b` — case split on kappa s vs kappa b

The third argument `void` does **not** match `.delta _`, so the second kappa equation applies:

$$\kappa(\texttt{recDelta } b\; s\; \texttt{void}) = \max(\max(\kappa b, \kappa s), 0) = \max(\kappa b, \kappa s)$$
$$\mu(\texttt{recDelta } b\; s\; \texttt{void}) = 3 + \mu b + \mu s > \mu b$$

## Case 3 — `R_rec_succ`: `recDelta b s (delta n) → app s (recDelta b s n)` — measure gap

This is the only case that **cannot be closed** by `drop_left` or `drop_right` without an additional helper lemma, and the gap is structural.

**Why kappa does not decrease in general.** The simp lemma `kappa_rec_delta` gives:
$$\kappa(\texttt{recDelta } b\; s\; (\texttt{delta } n)) = \max(\max(\kappa b, \kappa s), \kappa n) + 1$$

For the reduct, because `kappa (delta t) = kappa t`:
$$\kappa(\texttt{recDelta } b\; s\; n) = \begin{cases} \max(\max(\kappa b,\kappa s),\kappa m)+1 & \text{if } n = \texttt{delta } m \\ \max(\max(\kappa b,\kappa s),\kappa n) & \text{otherwise} \end{cases}$$

When $n = \texttt{delta } m$, then $\kappa n = \kappa m$, so both evaluate to $\max(\max(\kappa b,\kappa s),\kappa n)+1$. Kappa **does not decrease**.

**Why mu does not save us.** $\mu(\texttt{app } s\; (\texttt{recDelta } b\; s\; n)) = 4 + \mu b + 2\cdot\mu s + \mu n > 3 + \mu b + \mu s + \mu n = \mu(\texttt{recDelta } b\; s\; (\texttt{delta } n))$. Mu **strictly increases**.

**What is needed.** The case splits into two sub-cases based on whether `n` is delta-headed:

    -- kappa (recDelta b s (delta n)) = max(max(kb)(ks))(kn) + 1
    -- Strategy: show kappa drops, i.e.,

    -- which follows from (†) by omega, once (†) is established.

## Summary

| Rule | Strategy | Helper needed? |
|---|---|---|
| `R_int_delta` | `rcases` on `kappa t`; `drop_right` or `drop_left` | No |
| `R_rec_zero` | `rcases` on `kappa s` vs `kappa b`; `drop_right` or `drop_left` | No |
| `R_rec_succ` | `drop_left` after establishing `kappa (recDelta b s n) ≤ …` | **Yes — and the needed lemma is false under the current `kappa` definition** |
| `R_eq_refl` | `rcases` on `kappa a`; `drop_right` or `drop_left` | No |
| `R_eq_diff` | `drop_right` (kappa ties, mu drops by 1) | No |

Four of the five sorry cases are closable by the `drop_left`/`drop_right` pattern used in the proved cases. The `R_rec_succ` case exposes a genuine gap: `kappa` records only the presence (not the depth) of delta-wrapping on the third argument of `recDelta`, so it cannot distinguish `recDelta b s (delta n)` from `recDelta b s n` when `n` is itself delta-headed. Completing that case requires either replacing `kappa n` in the `recDelta(.delta n)` branch with a leading-delta counter, or adding a lexicographic third coordinate.
