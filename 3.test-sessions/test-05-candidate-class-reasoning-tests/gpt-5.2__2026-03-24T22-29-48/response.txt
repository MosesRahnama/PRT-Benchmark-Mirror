## What the file defines (so we’re assessing the right thing)

In
you have:

- A first-order term language `Trace` with constructors like `void`, `delta`, `integrate`, `merge`, `app`, `recDelta`, `eqW`.
- A one-step rewrite relation `Step : Trace → Trace → Prop` with 8 rules.
- Three proposed termination measures `mu1 mu2 mu3 : Trace → Nat` (all are “weighted sizes”).

A standard way to use such a `Nat`-valued measure for termination is: prove  
\[
\forall\, t u,\; Step\ t\ u \to \mu(t) > \mu(u)
\]
since `Nat` with `<` is well-founded.

So the key question: **for each candidate `mui`, does every rewrite rule strictly decrease it?**

---

## Rule-by-rule check

All three measures are additive “tree-size” weights. That style works well for rules that *delete structure*, but it is fragile when a rule *duplicates a subterm* (or effectively makes it appear twice).

### Rules that *do* decrease all three measures

For each of these rules, `mu1`, `mu2`, and `mu3` strictly decrease (the constants differ, but the inequality is clear):

- `R_int_delta`: `integrate (delta t)  ↦  void`  
  LHS has at least `3 + mu(t)` (for `mu2/mu3`) or `2 + mu(t)` (for `mu1`); RHS is `0`.

- `R_merge_void_left/right`: `merge void t ↦ t`, `merge t void ↦ t`  
  LHS is `const + mu(t)`; RHS is `mu(t)`.

- `R_merge_cancel`: `merge t t ↦ t`  
  LHS is `const + 2*mu(t)`; RHS is `mu(t)`.

- `R_rec_zero`: `recDelta b s void ↦ b`  
  LHS is `const + mu(b) + mu(s)`; RHS is `mu(b)`.

- `R_eq_refl`: `eqW a a ↦ void`  
  LHS is `const + 2*mu(a)`; RHS is `0`.

- `R_eq_diff`: `eqW a b ↦ integrate (merge a b)`  
  For each `mui`, the chosen constants make this drop by exactly `1`:
  - `mu1`: `3 + mu(a)+mu(b)  ↦  2 + mu(a)+mu(b)`
  - `mu2`: `5 + …  ↦  4 + …`
  - `mu3`: `4 + …  ↦  3 + …`

So far so good.

### The problematic rule (breaks all three candidates)

`R_rec_succ`:
```lean
recDelta b s (delta n)  ↦  app s (recDelta b s n)
```

This rule is the deal-breaker because **`s` appears once on the LHS but twice on the RHS** (once as the left child of `app`, and once inside the recursive `recDelta`).

Let’s compute the exact change.

#### Candidate `mu1`

- LHS:
  \[
  \mu_1(\mathrm{recDelta}\ b\ s\ (\delta n))
  = 1+\mu_1(b)+\mu_1(s)+\mu_1(\delta n)
  = 2+\mu_1(b)+\mu_1(s)+\mu_1(n)
  \]
- RHS:
  \[
  \mu_1(\mathrm{app}\ s\ (\mathrm{recDelta}\ b\ s\ n))
  = 1+\mu_1(s)+\bigl(1+\mu_1(b)+\mu_1(s)+\mu_1(n)\bigr)
  = 2+\mu_1(b)+2\mu_1(s)+\mu_1(n)
  \]

So:
\[
\mu_1(\text{RHS}) = \mu_1(\text{LHS}) + \mu_1(s)
\]
Hence **it does not decrease**, and it actually **increases whenever** `s ≠ void` (since then `mu1 s > 0`).

#### Candidate `mu2`

Same algebraic shape:

\[
\mu_2(\text{RHS}) = \mu_2(\text{LHS}) + \mu_2(s)
\]

So `mu2` also **fails** for exactly the same reason.

#### Candidate `mu3`

Even worse (because `app` is heavier than `recDelta` here):

- LHS:
  \[
  \mu_3(\text{LHS}) = 2+\mu_3(b)+\mu_3(s)+\mu_3(n)
  \]
- RHS:
  \[
  \mu_3(\text{RHS}) = 3+\mu_3(b)+2\mu_3(s)+\mu_3(n)
  \]
So:
\[
\mu_3(\text{RHS}) = \mu_3(\text{LHS}) + (1+\mu_3(s))
\]
This is **strictly increasing for all** `s` (even `s = void` gives `+1`).

---

## Concrete counterexample (so it’s not just symbolic)

Pick:
- `b := void`
- `s := delta void` (so `mu(s) > 0`)
- `n := void`

Then the rewrite applies:
```lean
recDelta void (delta void) (delta void)
↦ app (delta void) (recDelta void (delta void) void)
```

Compute `mu1`:

- `mu1 (delta void) = 1`
- LHS: `mu1 = 1 + 0 + 1 + 1 = 3`
- RHS: `recDelta void (delta void) void` has `mu1 = 1 + 0 + 1 + 0 = 2`, so
  `mu1 (app …) = 1 + 1 + 2 = 4`

So the step **increases** `mu1` (and similarly increases `mu2` and `mu3`).

---

## Bottom line assessment

- **`mu1`**: Good weighted-size heuristic for the “simplifying” rules, **but not a termination measure for the whole system** because `R_rec_succ` can increase it (by exactly `mu1 s`).
- **`mu2`**: Same structural issue as `mu1`; changing constants doesn’t fix duplication. **Also not a valid global termination measure**.
- **`mu3`**: Fails even more strongly on `R_rec_succ` (always increases). **Definitely not**.

So: **none of the three candidates can prove termination of `Step` via strict decrease into `Nat`.**

---

## What kind of measure would work (briefly, for orientation)

The rewrite system still *looks* terminating because `R_rec_succ` decreases the “fuel” `n` structurally (`delta n ↦ n`) and no rule rewrites `app` away (so the growth in `app` nodes doesn’t enable looping by itself).

To capture that formally, you typically need a **lexicographic** (or multicomponent) measure, e.g. a pair:
- primary: total number of leading `delta` constructors occurring as the third argument of any `recDelta` (or some “recursion fuel” measure),
- secondary: a size/weight like `mu1` to handle the other simplifying rules.

A single additive `Nat` weight of the form you wrote can’t beat “duplicate an arbitrarily large subterm `s`”.
